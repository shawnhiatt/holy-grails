# Holy Grails — Codebase, Performance & UX Audit

**Date:** July 2026 · **Version audited:** 0.5.6 · **Scope:** full codebase, sync architecture, followed users, user flows, visual layout, bundle/runtime performance.

> **Remediation status (same branch, commits P0–P4):** all priority items below have been implemented — §1.1/1.2 security fixes (`oauth.completeLogin`, internal `users.upsert`, 90-day session TTL), §3.2 zero-vinyl stall, §2.2 code splitting + vendor chunks, §3.4 server-side sync with adaptive throttling + `sync_status` progress, §4.2 followed-collections persistence (`followed_items`) with instant follow, and the §5/§7/§8 runtime & polish items (screen-local search, no keystroke remounts, `content-visibility`, `100dvh`, Fisher-Yates, token drift, dead code). The unused shadcn `ui/` directory and its 37 standby dependencies were subsequently removed as well (Figma Make is no longer part of the workflow). In a follow-up pass, the remaining items were closed as well: per-device `auth_sessions` table with rotation on every login (§1.2), SVGO on the logo assets + removal of the raster-in-SVG favicon (§2.2.4), font weights trimmed to 400–700 with preconnect + `<link>` loading (§2.2.5), a build-time guard for the missing-`VITE_CONVEX_URL` empty-shell trap (§2.3), `syncFailed` reset per attempt (§3.3), the duplicate profile memo (§4.3), and full removal of the dead crate/artwork view modes (§7, −1,000 lines). List virtualization (§5.2) is intentionally not implemented — `content-visibility: auto` covers current collection sizes; revisit only if collections exceed ~5k records.

This audit was requested as a full pass across every screen and interaction, with special attention to the recently reworked collection/wantlist/following sync and to the followed-users experience. Findings are ranked. Nothing in this document has been changed in code — it is a recommendation set.

---

## Executive summary

The app is in genuinely good shape. The recent sync rework (cache-first boot → cheap count probe → background diff sync → incremental Convex writes) is the right architecture and is well executed. List rendering is memoized, images are lazy-loaded, animations stick to transform/opacity, and the design system is applied consistently.

The audit found **one critical security hole**, **one user-facing stall bug**, and three structural issues that will matter as the app grows: a monolithic 1.22 MB JS bundle, a followed-users data layer that re-downloads entire collections from Discogs every session, and a single mega-context that makes every state change app-wide.

## 1. Security — critical

### 1.1 `users.upsert` allows account takeover (fix before anything else)

`users.upsert` is intentionally unauthenticated (it runs during OAuth callback, before a session exists) and takes `discogs_username` **from the client**:

- The Convex deployment URL ships in the client bundle, so anyone can call any public function.
- Calling `upsert({ discogs_username: "victim", access_token: "x", token_secret: "x" })` **returns the victim's existing `session_token`** (`convex/users.ts:96-105`).
- With that token the attacker reads/writes everything keyed to that user: purge tags, sessions, play history, preferences, following list, the cached collection and wantlist — and can call `users.deleteAllUserData`.
- As a side effect it also **overwrites the victim's stored Discogs OAuth tokens** with garbage, silently breaking their sync.

The session-token guard everywhere else is solid; this one entry point undoes it.

**Recommended fix (server-derived identity):**

1. Add an action `oauth.completeLogin({ oauth_token, oauth_verifier, token_secret })` in the `"use node"` OAuth file that:
   - performs the access-token exchange (existing `accessToken` logic),
   - calls `/oauth/identity` with the **received** access token to get the authoritative username,
   - calls an `internalMutation` (`users.upsertInternal`) with that server-derived username,
   - returns `{ username, avatarUrl, sessionToken, is_new }` to the client.
2. Convert `users.upsert` to an `internalMutation` (or delete it).
3. `auth-callback.tsx` then makes **one** action call instead of three (`exchangeToken` → `fetchIdentity` → `upsertUser`), which also removes a round trip from login.

This is a contained change: `auth-callback.tsx`, `convex/oauth.ts`, `convex/users.ts`. No UI changes. Deploy Convex before Vercel.

### 1.2 Session tokens never expire (medium)

`session_token` is a UUID minted once and reused forever (even re-used across logins). Combined with localStorage persistence, a leaked token is valid indefinitely. Cheap hardening, in order of value:

- Rotate the token on every OAuth login (currently explicitly reused).
- Add `session_created_at` and reject tokens older than e.g. 90 days in `authenticateUser`, forcing a re-login (the OAuth flow makes this painless).

### 1.3 Minor notes

- `getLatestUser` echoes `session_token` back in its payload. The caller already has it — drop the field so tokens exist in as few places as possible.
- `getPublicActivitySummary` and `getHolyGrailsUsers` correctly implement the shareActivity gate (authenticated viewer, opt-in check, indistinguishable `null`). Good pattern, keep it.
- OAuth PLAINTEXT signing for the transient handshake over HTTPS remains acceptable.

## 2. Bundle & load performance

### 2.1 The numbers

Production build (`vite build`, v0.5.6):

```
dist/assets/index.js    1,224.51 kB │ gzip: 324.34 kB   ← one chunk, everything
dist/assets/index.css     102.11 kB │ gzip:  17.37 kB
logo-holy-grails-splash    80.08 kB │ gzip:  33.18 kB
logo-holy-grails-dark      38.51 kB │ gzip:  15.62 kB
PWA precache: 13 entries (1,576 KiB)
```

There is **zero code splitting**. Every user — including a first-time visitor who only ever sees the splash screen — downloads and parses Recharts, the full album detail panel, reports, settings, folders, the add-albums drawer, etc., before React can render anything.

Verified contents: the heavy shadcn dependencies (`react-hook-form`, `react-day-picker`, `embla`, `cmdk`, `input-otp`, `react-resizable-panels`, `vaul`, most Radix packages) are **not** in the bundle — nothing outside `ui/` imports them, so they tree-shake away. The weight is: Recharts + d3 (~450 KB min, the largest single block), Motion (~110 KB), Convex client, React DOM, and ~300 KB of app code.

### 2.2 Recommendations (highest value first)

1. **Lazy-load `ReportsScreen`** — it is the only Recharts consumer. `const ReportsScreen = lazy(() => import("./components/reports-screen"))` with a `Suspense` fallback rendering the existing Disc3 spinner inside the screen container. One line of risk, removes ~450 KB (≈120 KB gzip) from the critical path. The `renderScreen()` switch in `App.tsx` is already the natural seam.
2. **Lazy-load the album detail panel** (`album-detail.tsx`, 3,166 lines) the same way. It's never needed for first paint. Keep `AlbumDetailSheet`/`AlbumDetailPanel` exports behind one `lazy` boundary in `App.tsx`.
3. **Vendor chunking** — add `build.rollupOptions.output.manualChunks` to split `react`/`react-dom`, `convex`, and `motion` into separate chunks. Today, any one-line app change invalidates the entire 1.2 MB precache entry on deploy; with vendor chunks, returning PWA users re-download only the app chunk (~300 KB). This matters because `vite-plugin-pwa` precaches everything.
4. **Splash logo SVG (80 KB)** — this is precached and loaded on the most latency-sensitive screen. Run it through SVGO; if it stays > 30 KB, the paths likely contain an embedded raster or excessive precision and it's worth re-exporting.
5. **Font loading** — `fonts.css` uses a blocking `@import` to Google Fonts (4 families, 10 weights). Move the stylesheet `<link>` plus `<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>` into `index.html` so the fetch starts before CSS parse, and audit whether all five weights of both families are actually used (each weight of Bricolage + DM Sans is a separate woff2). `display=swap` is already correct.
6. Nice-to-have: `<link rel="modulepreload">` hints come free once chunks are split; nothing to do manually.

Expected outcome of 1–3: first-load JS drops from ~324 KB gzip to roughly ~180 KB gzip, and repeat-visit downloads after a deploy drop by ~70%.

### 2.3 Build-environment footnote

Building without `VITE_CONVEX_URL` set silently produces a 176 KB bundle: Vite constant-folds the guard in `main.tsx` into an unconditional `throw`, and Rollup dead-code-eliminates the entire app. Harmless in production (Vercel sets the var) but a booby trap for CI or local builds — a size-regression check would pass while shipping an empty app. Consider failing the build explicitly (`if (!process.env.VITE_CONVEX_URL) throw` in `vite.config.ts`) rather than at runtime.

## 3. Sync architecture (the recent rework) — review

### 3.1 What's working — keep all of this

- **Cache-first boot**: hydrate from Convex unconditionally, probe in the background. This is the correct shape and the code reads well.
- **`proxyFetchSyncSignals` count probe**: turning the "nothing changed" case into one request is exactly right, and persisting the raw pre-filter counts (so they compare against `num_collection`) shows attention to detail.
- **`applyDiff` incremental writes**: signature comparison per row, no empty-cache flash, and the added/removed counts driving the toast is a nice touch.
- **`authedArgs` stale-token gate**, the in-flight ref guards (`bgSyncInFlightRef`, `followingFeedSyncInFlightRef`), and the throttled sync-on-focus probe are all sound.
- **`SyncStatusLine`** as a manual probe trigger with "Up to date." feedback closes the loop the silent paths leave open.

### 3.2 Bug: zero-vinyl user is stuck on the loading screen (high)

`isAuthLoading` (`app-context.tsx:376`):

```ts
const isAuthLoading = (!!discogsUsername || isRestoringSession) && albums.length === 0 && !isConvexUserGone && !syncFailed;
```

For a user whose Discogs collection is empty — or contains **no vinyl** (all CDs/cassettes, stripped by the vinyl filter) — `albums.length` stays 0 after a *successful* sync, so `isAuthLoading` stays `true` and `App.tsx` renders `LoadingScreen` forever. Every new user you onboard who hasn't bought vinyl yet hits this.

**Fix:** track sync completion explicitly instead of inferring it from data presence — e.g. a `hasCompletedFirstLoad` flag set when the boot effect finishes (cache hydrated *or* `performSync` resolved/rejected), and gate `isAuthLoading` on that. The empty Collection screen already has good empty-state copy waiting behind it.

### 3.3 Correctness notes (medium/low)

- **Crossover queue only populates on full syncs.** `collectionCrossoverQueue` is set in `performSync`, but the common boot path (cache hydration, probe says "unchanged") never computes crossovers — so a wantlist item bought and synced on another device won't trigger the prompt until counts change. If intended, fine; otherwise compute crossovers during cache hydration too (both datasets are in hand).
- **Count probe can't see in-place edits** (condition/notes edits on Discogs.com, or simultaneous +1/−1). Already documented; manual Sync Now covers it. Optional improvement: compare `num_collection` **and** the newest `date_added` from page 1 — still cheap, catches swap edits.
- **`performSync` foreground error path**: on first-ever sync failure, `syncFailed` short-circuits `isAuthLoading` and the toast points to Settings — good. But the background path (`maybeBackgroundSync`) swallows the error after `setSyncFailed(true)`, and `syncFailed` is never reset by a later successful background sync (only `syncFromDiscogs` resets it). Low impact, worth a `setSyncFailed(false)` at the top of `performSync`.
- **`loginWithOAuth` stores nothing if the tab closes mid-sync** — the sync is awaited but the session token was already persisted, so recovery on next load works. Fine.

### 3.4 Sync speed & data volume (medium)

Two structural costs in the current pipeline:

1. **Fixed 1.1 s sleep between every collection page** (`convex/discogs.ts:557`). Discogs returns `X-Discogs-Ratelimit-Remaining` on every response. An adaptive throttle — full speed while remaining > 10, back off as it approaches 0 — would cut a 1,000-record sync from ~15 s of pure sleeping to a few seconds, and would make the multi-user following-feed sync dramatically faster too. The 429 retry you already have stays as the backstop.
2. **Every synced byte crosses the wire twice.** The action fetches from Discogs → returns the full collection to the browser → the browser uploads it all back to `applyDiff`. For a 2,000-record collection with custom fields, that's a multi-megabyte round trip on every changed sync (and Convex function arguments have size limits you'll eventually brush against).

**Recommendation — move the sync loop server-side.** Have a single action `syncCollection(sessionToken)` that fetches pages and calls `internal` mutations to apply diffs as it goes, writing progress (`page X of Y`, phase) to a small `sync_status` document. The client:

- subscribes to `sync_status` via `useQuery` → **this restores the per-page progress feedback you lost in the 1b migration** (currently a documented backlog item) with no callback plumbing;
- receives the fresh collection reactively through the existing `convexCollection` subscription instead of a giant action return value.

This is the single highest-leverage refactor in the sync layer: less bandwidth, real progress UI, survives tab close mid-sync, and shrinks `performSync` in `app-context.tsx` substantially. It's a "flag before refactoring" file, so treat this as the flag.

## 4. Followed users — the area that "could use some love"

This is the weakest data layer in the app, and the problems compound with each user followed.

### 4.1 Current behavior and costs

1. **Full re-hydration every session** (`app-context.tsx:819-888`). First visit to the Following screen kicks off a sequential loop: for *every* followed user — profile fetch, full paginated collection fetch, full wantlist fetch, 1 s between users. For 8 followed users with ~800-record collections that is **~100 Discogs requests and 2–4 minutes** of background fetching, burning most of the 60/min rate budget… and the result is held in React state only. Kill the tab, pay again tomorrow.
2. **Follow is blocking** (`following-screen.tsx handleConnect`). "Connect" awaits the full collection + wantlist fetch before the follow registers. Following someone with 2,000 records means watching "Fetching collection..." for 30+ seconds. Meanwhile `addFollowedUser` *also* fetches page-1 data for the feed cache — duplicate work.
3. **Two parallel data models.** `followedUsers[].collection/wants` (full, memory-only) vs `followingFeed` (50 recent, Convex-cached). The Feed uses the cache; the profile view uses the full copy; avatars live in a third place (`following.avatar_url`). The `hydrated` flag and `refreshFollowedUser` recovery path exist to patch over silent hydration failures — a symptom of the split.
4. **Comparison filters need the full data.** "In Common", "They Want / You Cut", "You Want / They Have" are the genuinely differentiating features here, and they're why the full fetch exists.

### 4.2 Recommended target architecture

Mirror the pattern you already built for the user's own data — it works:

1. **Persist followed collections in Convex** — a `followed_collections` table (or extend `following_feed`) storing the mapped album rows per followed user, with `lastSyncedAt`. Slim rows are fine: release_id, master_id, title, artist, year, thumb, cover, label, dateAdded — the profile view uses nothing else (`skipPrivateFields` already strips conditions/folders).
2. **Hydrate per-profile, on demand, cache-first.** Opening a followed user's profile: render instantly from Convex, run the count probe (`num_collection` from their public profile — the same `proxyFetchSyncSignals` trick works for any username), re-fetch only if changed or stale. Kill the "hydrate everyone on screen open" loop entirely.
3. **Make follow instant.** `handleConnect`: profile lookup (1 request) → follow recorded, page-1 feed data fetched (already happens) → toast. The full collection syncs in the background or on first profile open, with the existing partial-entry skeleton UX.
4. **Sync followed users server-side with the adaptive throttle** from §3.4 — a `syncFollowedUser(username)` action writing straight to Convex. The client's only job becomes subscribing.
5. **Unify on one model**: `followingFeed` remains the "recent activity" projection; the profile reads the persisted full collection. Delete the `hydrated`/`refreshFollowedUser` recovery machinery once hydration is no longer a fragile session-long loop.

Payoff: Following screen opens instantly every session, rate-limit budget is freed for your own syncs, comparison filters work offline, and follows feel instant. This is the biggest UX improvement available in the app right now.

### 4.3 Smaller following-screen notes

- The 24 h TTL bypass conditions (`needsMasterIdMigration`, `needsWantsMigration`) are one-time migrations that now run on every sync pass for users whose caches predate the fields — they self-heal, fine, but worth deleting once all real caches have migrated.
- `syncFollowingFeed` fires collection + wantlist page fetches **in parallel per user** with 1 s between users → ~2 req/s sustained for 25 users, right at the 60/min limit if a collection sync is also running. The server-side rewrite with header-aware throttling resolves this properly.
- `FollowedUserProfile` builds `byReleaseId` maps twice (recentlyPlayed and topPlayed memos) — merge into one memo. Cosmetic.

## 5. Runtime rendering performance

### 5.1 The mega-context (medium — the main runtime cost)

`AppState` carries ~90 fields; 46 `useApp()` call sites. The context `value` is one `useMemo` whose deps include high-frequency state (`searchQuery`, `filteredAlbums`, `showAlbumDetail`, header callbacks…). Any change — every search keystroke — creates a new context value and re-renders **every** consumer: both navs, install nudge, offline banner, sheets, the active screen, everything. The memoized leaf cards (`GridCard`, `ListRow`, `RecentAlbumCard`) contain the damage well, which is why the app still feels fine — but each keystroke still re-runs the full filter+sort over `albums` (O(n log n)) and re-renders every screen-level component.

Recommendations, cheapest first:

1. **Move `searchQuery`/`wantSearchQuery` out of the context.** They're screen-local concerns; `crate-browser` and `wantlist` can own them (the filter drawer can receive them via props). This alone removes the per-keystroke app-wide invalidation. `filteredAlbums` moves with it (or becomes a `useFilteredAlbums(albums, opts)` hook).
2. **Split the context in two**: `AppDataContext` (albums, wants, stacks, feed — changes rarely) and `AppUiContext` (screen, panels, selections — changes often). Consumers pick one. This is an internal refactor; the `useApp()` facade can remain as a compatibility shim so components don't change, honoring the "public API of app-context must not change" rule.
3. Defer anything further (selector libraries, `use-context-selector`) until 1–2 prove insufficient — they likely will be sufficient at this scale.

### 5.2 Large-collection rendering (medium)

`AlbumGrid`/`AlbumList` render **all** filtered albums — a 1,500-record collection is ~1,500 mounted cards with images. Lazy image loading keeps network sane, but DOM size makes initial screen render and theme toggles sluggish on mid-range phones.

Cheapest meaningful fix: add `content-visibility: auto; contain-intrinsic-size: <card estimate>` to the card wrappers (`GridCard` container, `ListRow`). The browser skips layout/paint for offscreen cards — most of virtualization's benefit, zero architectural change, and it doesn't disturb the alphabet-sidebar anchor refs. Real windowing (`@tanstack/react-virtual`) only if collections in the wild exceed ~3–5 k; note it *would* complicate the alphabet index and grid dividers, so don't reach for it prematurely.

### 5.3 Small things (low)

- `sort(() => Math.random() - 0.5)` (feed shuffle spots, ×3) is a biased shuffle. A 5-line Fisher-Yates in a shared util fixes it; also replaces the double-weighting hack in `huntAlbums` awkwardness.
- Feed section picks (`decadesSpotlight`, `depthsAlbums`, `huntAlbums`) are `useState` initializers — correct for per-mount stability, but note they intentionally never update on background-sync changes; fine, just deliberate.
- `buildPlayMaps` ISO-string comparison for "most recent" works but comparing raw `played_at` numbers is simpler and cheaper — do it when next touching the file.
- `DominantColorCard` is well done (50×50 downsample, module cache, stale guard). One gap: extraction failures aren't cached, so a broken image retries on every mount. Cache a null result too.
- `SyncStatusLine`'s 60 s interval ticker is fine (one small component re-render).

## 6. User flows & UX findings

Screen-by-screen pass. Only friction points listed — the flows generally hang together well.

### Boot & auth
- **Zero-vinyl stall** — see §3.2. This is the worst first-run outcome possible (new user, permanent spinner).
- OAuth abandonment detection (visibilitychange + `oauthInFlight`) is thoughtful and works for the back-out case.
- After the security fix in §1.1, login gets one round trip faster for free.

### Feed
- Section picks stable per mount — good call, prevents content shifting under the user's thumb.
- The purge evaluator inline card + crossfade is a strong retention mechanic.
- `getCachedCollectionValue()` returns null until the async value fetch resolves; the section handles it. Fine.

### Collection
- Search forcing artist-A→Z with `effectiveSortOption` is good, predictable behavior.
- Per-keystroke cost: see §5.1. With `content-visibility` and local search state, search will feel instant even at 3 k records.
- `SyncStatusLine` placement (under search row) reads well.

### Wantlist / crossover
- Crossover prompt only fires on full syncs (§3.3) — verify that's intended.
- Add/remove wantlist flows (API-first, then state, then cache) are the right order and consistently applied everywhere, including the heart button variants.

### Purge
- `executePurgeCut` re-runs a **full** `syncFromDiscogs()` after the loop even though each deletion already updated local state + Convex individually. With the diff sync this is cheap-ish but still a full paginated fetch; consider dropping the re-sync (the local state is already authoritative — deletions were confirmed per-item) or replacing it with the count probe.
- 1 s between deletes is safe; with header-aware throttling (§3.4) large cut batches would finish much faster.

### Following
- See §4 — the big one.
- Unfollow confirmation flow (header button → modal → `removeFollowedUser`) is coherent; feed cache cleanup on unfollow is handled.

### Settings / folders / profile
- Folder operations wait for API success without optimistic updates — correct for destructive/remote-authoritative ops.
- `wipeAllData` ordering (Convex delete while token valid, then local reset) is correct.

### Offline
- `OfflineBanner` + probe returning `null` on network failure degrade gracefully. Cached boot means the app is genuinely usable offline. Good PWA behavior.

## 7. Visual & layout consistency (code-level pass)

- **`h-screen` on the app root** (`App.tsx:380`). CLAUDE.md's own rule says never use `100vh` for full-screen layouts on iOS Safari. In-browser (pre-install) usage will size the root behind Safari's chrome — the bottom tab bar can sit partially behind the toolbar until the user installs the PWA. Change to `style={{ height: "100dvh" }}` (splash/loading screens already do this correctly). The remaining `calc(100vh - 58px)` maxHeights in the sheets should become `100dvh` at the same time.
- **Border-color drift**: `album-grid.tsx:70` uses `#D1D8DF` for the light-mode card border where `feed-screen.tsx`/`crate` cards use `#D2D8DE` (the `--c-border` token). One-character drift, but it's exactly the kind the Oklab rules exist to prevent — and both should arguably be `var(--c-border)` anyway. (`#D1D8DF` in `navigation.tsx` is the documented fixed nav color — leave it.)
- **Dead `ViewMode` values**: the type still declares `"crate" | "artwork" | "grid3"`; crate/artwork were removed from the product (guard-reset on mount per CLAUDE.md) but `grid3` is used only inside the following profile's local toggle. Narrow the shared type to `"grid" | "list"` and give the following profile its own local type — prevents removed modes from resurfacing via stored prefs.
- **Z-index**: all values encountered match the documented hierarchy table. No conflicts found.
- **Safe areas, 16 px inputs, inline truncation styles**: consistently applied everywhere sampled (feed cards, grid, list rows, following rows, sheets). The discipline here is unusually good.
- Empty-state inconsistency is already on your backlog; confirmed it's real (icon sizes 48 px vs paddings varying) but per instructions, leaving for the dedicated design pass.

## 8. Code health & structure

- **`app-context.tsx` (2,571 lines) is doing ~10 jobs**: auth/session, theme, preferences, collection state, wantlist, sessions, plays, following, two sync engines, header callback plumbing. It's load-bearing and flagged, so: recommend extracting *implementation* into hooks (`useSyncEngine`, `useFollowing`, `useStacks`, `usePlays`) that the provider composes, keeping `useApp()`'s shape identical. Do it opportunistically — e.g., the following rework (§4) naturally pulls that domain out.
- **`feed-screen.tsx` duplication**: `buildFeedActivity` / `buildFeedWantActivity` are the same function modulo the source array and id prefix — merge with a parameter.
- **Boot-hydration duplication**: the cache-hydration block in the boot effect re-implements the purge-tag/priority/play merging that `performSync` also does. A shared `mergeConvexOverlays(albums, wants)` helper would halve that effect and remove a class of drift bugs.
- **Unused dependencies**: nothing imports `src/app/components/ui/` at all. The 15+ Radix packages, `react-hook-form`, `react-day-picker`, `embla-carousel-react`, `cmdk`, `input-otp`, `react-resizable-panels`, `vaul`, `next-themes`, `react-resizable-panels` are dead weight in `node_modules`/install time (they tree-shake out of the bundle). Either delete the unused `ui/` components + deps, or accept them consciously as a component library on standby. `recharts`'s `ui/chart.tsx` wrapper is also unused (reports imports recharts directly).
- **`wantlist.replaceAll`** is still used for the empty-cache fallback path; `collection.replaceAll` appears fully superseded by `applyDiff` — verify and delete.
- **Duplicate commit history noise** (`58a208f`/`2e39a3f`, `ee39a02`/`7f21a04` pairs) — cosmetic, but suggests a double-push habit worth watching.
- Console hygiene is excellent (zero `console.log`; warns/errors only and purposeful).

## 9. Suggested sequencing

Each phase is one focused session, per your working rules:

1. **P0 — Security**: server-derived identity for `users.upsert` (§1.1) + drop token echo (§1.3). Convex deploy required.
2. **P1a — Bug**: zero-vinyl loading-screen stall (§3.2).
3. **P1b — Bundle**: lazy `ReportsScreen` + lazy album detail + vendor `manualChunks` (§2.2 items 1–3). Measure before/after with `vite build`.
4. **P2 — Followed users rework** (§4.2): persist followed collections in Convex, on-demand per-profile hydration, instant follow. Biggest UX win; also the natural moment to extract the following domain from `app-context.tsx`.
5. **P3 — Server-side sync loop** (§3.4): adaptive throttle + progress doc; restores the sync-progress backlog item.
6. **P4 — Runtime polish**: search state out of context / context split (§5.1), `content-visibility` on cards (§5.2), `100dvh` root (§7), border-token fix, Fisher-Yates util, dead-code deletion (§8).
