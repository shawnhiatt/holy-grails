## Spec 3 — Share a Session

**The use case (from Shawn):** house party; only Shawn has HG; the designated DJ gets
texted the session and needs it to open on any phone with zero install/login.

**Recommended shape — a public read-only share link, plus plain-text fallback:**

A URL wins over a text blob for the DJ case: artwork, order, readable on any device, and
it stays live if the session is edited before the party. Text export rides along nearly
free via the share sheet.

### 3a. Backend — capability-token share links

- `convex/schema.ts` — add to the `stacks` table: `share_id: v.optional(v.string())`
  plus `.index("by_share_id", ["share_id"])`. No migration needed (optional field).
- `convex/stacks.ts`:
  - `enableShare` (mutation, authed via `authenticateUser`): generates an unguessable
    `share_id` (crypto-random, ≥128 bits — e.g. 22+ chars base62; `crypto.randomUUID()`
    with dashes stripped is acceptable), patches the stack, returns it. Idempotent —
    returns the existing `share_id` if already set.
  - `disableShare` (mutation, authed): clears `share_id`. This is revocation.
  - `getShared` (query, **deliberately unauthenticated** — the unguessable `share_id` IS
    the capability, same philosophy as the Cross-User Data Pattern's "null for not-found
    and not-shared alike"): looks up by `by_share_id` index; returns `null` if not found.
    Returns ONLY: session `name`, `last_modified`, and an ordered album array of
    `{ title, artist, year, cover, thumb }`. Resolve albums server-side by joining
    `album_ids` against the sharer's `collection` cache rows
    (`by_username_and_release`). **Never** return `discogs_username`, release ids you
    don't need, purge tags, notes, conditions, price paid, or any token. Albums that have
    left the collection since sharing are silently skipped.
- **Tests are mandatory** (CLAUDE.md testing rule — new cross-user surface): a new
  `convex/stacks.test.ts` (edge-runtime docblock first line) covering: `getShared` with a
  valid share_id returns only the whitelisted fields; unknown share_id → `null`; revoked
  (disabled) share → `null`; `enableShare`/`disableShare` reject bad session tokens; the
  returned payload never contains `discogs_username`/tokens/notes/conditions.

### 3b. Client — the share page

- Route: `holygrails.app/s/{shareId}`. `App.tsx` already branches on pathname for the
  OAuth callback — add a pre-auth branch: if `location.pathname.startsWith("/s/")`,
  render a new `SharedSessionPage` component INSTEAD of the splash/auth flow. No login
  required. (Also add the rewrite to `vercel.json` if SPA rewrites aren't already
  catch-all — check first.)
- `src/app/components/shared-session-page.tsx` (new): mobile-first read-only page —
  HG wordmark small at top (light/dark aware), session name in Bricolage Grotesque,
  "{N} records", then a simple numbered list: thumb (56px, use `thumb || cover`), title,
  artist · year. Follow the design system: tokens only, DM Sans body, no new colors,
  `--nav-clearance` not needed (no nav), safe-area padding at the bottom. Footer line:
  "Shared from Holy Grails." — no signup CTA, no discogs.com links (absolute rule).
  This page must work logged-out in an incognito tab.
- Dark/light: default to system (`prefers-color-scheme`), since there's no preference
  context for an anonymous viewer.

### 3c. Client — the share affordance

- In `stacks.tsx`, add a Share action per session (icon button in the session header/row —
  match existing inline-edit affordance patterns in that file; icon via `icons.ts` shim,
  e.g. Phosphor's ShareFat/Export under a legacy alias — add the alias to `icons.ts`).
- Tap → calls `enableShare`, gets the URL, then:
  - If `navigator.share` exists (iOS/Android): share sheet with
    `{ title: name, url }` — this covers "text it to the DJ" natively.
  - Else: copy to clipboard + toast `Link copied.`
- Add "Stop sharing" for an already-shared session (calls `disableShare`, toast
  `Sharing stopped.`). Keep the UI minimal — an accordion row or a second tap-state, match
  the file's existing patterns.
- Optional-but-cheap text fallback: a "Copy as text" secondary action producing
  `Session name\n1. Artist – Title\n2. …` for maximal-compat texting.

**Toasts:** `Link copied.` / `Sharing stopped.` — under 4 words, per UX rules.

**CLAUDE.md updates:** Data Architecture (stacks table gains `share_id`; note the
capability-token pattern and that `getShared` is intentionally unauthenticated), File
Structure (new component), Testing section (new test file), and the Sessions naming note
still applies — all new identifiers stay `stack*`/`share*`, user-facing copy says
"Session".

**Acceptance:** share link opens logged-out on another device and renders the list;
revoke kills the link (shows a simple "This session is no longer shared." state);
Convex tests pass; `npx convex deploy` run; typecheck/lint/test green.

## Spec 4 — Insights improvements, phase 1 (zero new API calls)

### Objective review of the current Insights screen

**What's working (keep all of it):**
- Threshold-gating with `return null` instead of empty states is the single best design
  decision on the screen — sparse collections never see weak content. Every new section
  below must follow it.
- Condition spectrum, purge donut, and the decade chart are genuinely collection-specific
  insights Discogs doesn't give you. The "NM or better" callout is the model: one derived,
  opinionated sentence per section.
- Listening Activity is differentiated (Discogs has nothing like it).

**Honest gaps:**
1. **Nothing on the screen moves.** Every section is a snapshot of composition (decade,
   format, label, artist). The most interesting story a collection tells is *trajectory* —
   when you collect, how fast, what a year of collecting looked like. `dateAdded` is
   sitting on every album row, unused.
2. **Money is underused.** There's a hero median value (one number from Discogs), but
   `pricePaid` is on every collection row and never aggregated. Spend-vs-value is the
   question every collector actually has. (Full per-album value needs Spec 6; spend-side
   works today.)
3. **No cross-referencing.** Sections never talk to each other — e.g. purge status ×
   value ("your Cut pile is worth $X") or plays × decade. Cheap derivations, high interest.
4. **Genre/style is absent** — the one axis every music-collection tool leads with. Not
   currently stored on collection rows (would need a sync-time schema addition; Discogs'
   `basic_information` includes `genres`/`styles`, so it's a slim-field add + resync —
   flagged as **phase 2**, its own session, since it touches `convex/discogs.ts` sync
   mapping + schema).

### Phase 1 scope (this session — no new data, no schema changes)

Add two sections and one callout, all derived from existing context data, all

## Spec 5 — Multiple accounts (Reddit-style switcher)

**Use case:** one Discogs account for 12″ collection, another for 45s. Rare but real.
Sign in once per account, then toggle.

### Architecture decision (made — don't relitigate in-session)

**All multi-account state is client-side.** The backend already supports this perfectly:
every login mints an independent `auth_sessions` row (per-device token rotation), sessions
are keyed by token, and nothing server-side assumes one-account-per-device. **Zero Convex
changes.** An "account" on a device is simply a stored session token.

**Switching = swap token + full reload.** `location.reload()` after swapping the active
token in localStorage. Rationale: `app-context.tsx` is 2,300 lines of hydration refs,
one-time effects, subscription gating, and sign-out edge cases (`hasSignedOutRef`,
`hydratedRef`, `initialSyncDoneRef`, ~30 state resets). An in-place switch must replicate
the sign-out reset *minus* the Convex session clear, *plus* re-arming every hydration
flag — high blast radius in the most load-bearing file in the app for a "more rare than
common" feature. A reload boots from the Convex cache subscriptions in ~1s and is
guaranteed-clean. If switch speed ever matters, in-place switching is a later
optimization with this storage model unchanged.

### Storage model

- New localStorage key `hg_accounts`: JSON array of
  `{ username: string, avatarUrl: string, sessionToken: string, addedAt: number }`.
- `hg_session_token` stays exactly as-is (the ACTIVE account's token) — full backward
  compatibility: existing signed-in devices just have an `hg_accounts` of length 0/1
  after a one-time migration on load (if `hg_session_token` exists and `hg_accounts`
  doesn't, seed it once username/avatar are known).
- All reads/writes wrapped in try/catch like the existing token persistence.

### Flows

1. **Add account** (Settings → Account card → "Add account" row): kicks off the normal
   OAuth flow (`oauth-helpers.ts` unchanged). In `loginWithOAuth`
   (`app-context.tsx:1715`), instead of implicitly replacing identity: upsert
   `{ username, avatarUrl, sessionToken }` into `hg_accounts` (dedupe by username —
   re-adding an account replaces its token), then proceed as today. This same upsert also
   runs on every normal login, which is what seeds the list.
   - **Discogs-side caveat to surface in UI:** the Discogs authorize page will
     auto-authorize whichever account is signed in at discogs.com in that browser. Copy
     under the button (12px, `--c-text-muted`): "You'll sign in on Discogs — use the
     account you want to add." Users switching Discogs accounts do it on Discogs' page.
2. **Switch** (Settings → Account card: list all accounts, active one marked, others
   tappable): write the target token to `hg_session_token`, then `location.reload()`.
   Show a brief pressed/loading state before reload fires.
3. **Sign out** (existing button): remove the ACTIVE account from `hg_accounts` and call
   `clearSessionMut` as today. If other accounts remain → promote the first remaining one
   (write its token, reload). If none remain → current behavior (login screen).
4. **Expired/invalid stored account:** the session-restore path already clears a dead
   `hg_session_token` (`app-context.tsx:497-506`). Extend: when the active token proves
   invalid, also remove that account from `hg_accounts`, and if others exist, promote the
   next instead of dumping to login. A *background* (non-active) account's expiry is
   discovered only when switching to it — same handling applies at that moment (toast
   `Session expired. Sign in again.`, remove it, stay on current account).
5. **Wipe all data** (`wipeAllData`): removes the active account from `hg_accounts` too.

### UI

Settings screen, Account card (`settings-screen.tsx`, above the Sign out button): a
divider, then an "Accounts" mini-list — each row: 28px avatar (initial fallback via
`getInitial`), username, active row gets a filled check in `var(--c-link)`; inactive rows
tappable (full-row button, `aria-label="Switch to {username}"`). Below: "Add account" row
with a `UserPlus`-style icon (reuse existing alias or add to `icons.ts`). Match the
card's existing row typography (14px/500 DM Sans). No header/avatar switcher in v1 —
Settings only, keep it discoverable but boring.

### Guardrail housekeeping (required, will fail lint otherwise)

- `eslint.config.js`: the localStorage whitelist message (line ~78) and the
  `app-context.tsx` override already permit app-context localStorage — if any account
  logic lands outside `app-context.tsx`, keep it IN app-context instead (all
  account-list read/write goes through context helpers; settings-screen only calls
  context functions). Update the rule's message string to mention `hg_accounts`.
- CLAUDE.md: the "localStorage is permitted in two places" section becomes three keys
  (`hg_session_token`, `hg_accounts` — both app-context — and the install-nudge flag);
  update the Authentication Architecture section with the multi-account model and the
  reload-based switch rationale.

### Testing

No new Convex functions → no new Convex tests required. Extract the accounts-list
manipulation (upsert/dedupe/remove/promote-next) as a pure module
(`src/app/utils/accounts.ts` or similar) with node Vitest coverage: upsert dedupes by
username, remove-active promotes next, corrupt JSON in storage → empty list (defensive
parse).

**Acceptance:** add a second account via OAuth; both listed in Settings; switching
reloads into the other collection (verify albums/wantlist/sessions/purge tags all belong
to the switched-to user); sign out of one leaves the other signed in; typecheck/lint/test
green.

## Spec 6 — Per-album market value, slow-drip (revival of the tabled note)

The tabled architecture note is sound — Convex cron + daily batch is the right call, and
the codebase has grown the exact primitives it was missing: adaptive rate limiting in
`discogsFetch`, the `"use node"` action/helper split, and a server-side sync-loop pattern
to copy (`syncSelf`). The old note's open decisions are now **decided**:

| Decision | Call |
|---|---|
| Endpoint | `/marketplace/stats/{release_id}` → store `lowest_price` (the note's choice; matches the Value section's "lowest ask" language) |
| Staleness | 30 days before re-fetch |
| Null handling | Store `null` + timestamp (so we don't hammer no-listing releases daily); **exclude** from rankings; no "no data" rows in top-5 lists |
| Scope | Current user's own collection only. Followed users: never in this phase |
| Batch size | 40/day per user |
| Visibility | "Values updated {relative time}" line via `formatSyncedAgo`, shown in the Insights value section |
| Manual kick | **Not in v1.** The "Refresh Now big batch" idea competes with the user's own 60/min budget while they might be syncing — revisit only if the ~2-week fill proves annoying in practice |

### Session A — backend

1. `convex/schema.ts` — `collection` table gains `marketValue: v.optional(v.number())`
   (null stored as… note: Convex optional + `v.union(v.number(), v.null())` — use
   `v.optional(v.union(v.number(), v.null()))` so "fetched, no listings" (null) is
   distinguishable from "never fetched" (undefined)) and
   `marketValueFetchedAt: v.optional(v.number())`. `users` table gains
   `market_cursor: v.optional(v.number())` (releaseId watermark, resumable).
2. `convex/crons.ts` (new) — daily cron (pick a low-traffic hour, e.g. 09:00 UTC)
   calling an internal action `discogs.marketValueDrip`.
3. `discogs.marketValueDrip` (internalAction in `convex/discogs.ts`, `"use node"`):
   - Iterate users that have valid OAuth creds (an internalQuery in `discogsHelpers.ts`
     lists them — remember queries can't live in the "use node" file).
   - Per user: query up to 40 collection rows ordered by releaseId above
     `market_cursor` whose `marketValueFetchedAt` is missing or >30 days old; fetch
     stats via the existing `discogsFetch` (adaptive limiter already handles pacing —
     do NOT add fixed sleeps); patch rows via an internalMutation
     (`collection.setMarketValue` — releaseId + username keyed, `by_username_and_release`
     index); advance/wrap `market_cursor` (wrap to 0 when the end is passed).
   - Per-user try/catch: one user's revoked token must not kill the whole run.
   - Currency note: `lowest_price` from stats comes as `{ value, currency }` — store
     `value` and assume the user's currency as Discogs returns it; if mixed currencies
     show up in practice, that's a fast-follow, don't solve it speculatively.
4. Reactive derivation: `app-context.tsx` already re-derives `albums` from `collection`
   cache subscriptions — decide whether `marketValue` rides onto the `Album` type now or
   waits for Session B; putting it on the type in Session A (typed, unused) keeps
   Session B UI-only. **Important:** confirm the sync `applyDiff` doesn't clobber
   `marketValue` fields on unchanged rows (the sync writes full row shapes — the diff
   writer must preserve the two market fields when patching/replacing rows it owns).
   This is the one genuinely fiddly integration point in the whole spec — test it.
5. Tests: `convex` tests for the cursor advance/wraparound and staleness selection logic
   (pure-function-extract the row-selection if convex-test can't drive the node action),
   and for `setMarketValue` auth (internal-only).
6. `npx convex deploy` (both env vars already exist; crons deploy with it).

## Spec 7 — Spotify / Apple Music playlist export: **defer** (assessment)

Shawn asked directly whether this is out of reach. Honest answer: **Spotify is feasible;
Apple Music is materially harder; neither belongs in this batch.** This was attempted
before and abandoned as complex — the complexity assessment was correct.

**Why it's a project, not a feature:**
1. **Two OAuth systems.** Spotify supports Authorization Code + PKCE from a pure client
   (no secret; a Settings-screen "Connect Spotify" is architecturally clean — token
   storage would go in Convex like Discogs creds). Apple Music (MusicKit JS) additionally
   requires a paid Apple Developer membership and a server-minted developer JWT
   (Convex-signable, but real key management) before any user auth happens.
2. **The matching problem is the real work.** A session is a list of *vinyl releases*;
   a playlist is a list of *streaming tracks*. Per album: search the streaming catalog by
   artist+title (normalize the Discogs artist-disambiguation suffixes), pick the right
   album from ambiguous results (remasters, deluxe editions, regional variants,
   compilations), fetch its track list, append. Expect a meaningful miss/mismatch rate
   and the need for a review-before-create UI ("12 of 15 matched — review") — that UI is
   most of the feature.
3. **Scope creep risk** into "streaming integration" (playback links, Listen On
   expansion), which the app has deliberately kept to two dumb brand-icon buttons.

**Recommendation:** park it. If/when it's picked up: Spotify only, as a dedicated
multi-session project with its own plan doc (`docs/spotify-export-plan.md`), phased as
(1) PKCE connect + token storage in Convex, (2) match engine + review UI, (3) playlist
creation. Apple Music only if the Spotify version earns it. The session-share page
(Spec 3) covers a chunk of the underlying want — getting a session out of the app and
into the room — at ~5% of the cost.
